# Data Model: E2E Integration Testing

**Feature**: 001-e2e-android-testing  
**Date**: 2026-04-05

## Entities

### TestScenario

Represents a single testable acceptance scenario from the spec.

| Field | Description |
|-------|-------------|
| id | Unique identifier (e.g., `US1-AS1`, `US4-AS2`) |
| user_story | Reference to parent user story (1-7) |
| priority | P1, P2, or P3 (inherited from user story) |
| title | Human-readable scenario name |
| preconditions | Required app/infrastructure state before test |
| steps | Ordered list of agent actions (navigate, tap, verify, etc.) |
| expected_outcome | What the agent should observe after steps complete |
| status | pending, running, passed, failed, skipped |
| duration_ms | Execution time in milliseconds |
| screenshots | List of screenshot paths captured during execution |
| failure_detail | Error description if status is failed |

### TestRun

Represents a single execution of the full E2E suite.

| Field | Description |
|-------|-------------|
| id | Unique run identifier (timestamp-based) |
| started_at | ISO 8601 timestamp |
| completed_at | ISO 8601 timestamp |
| mode | `ci` (pass/fail assertions) or `local` (explore-fix-verify) |
| scenarios | List of TestScenario results |
| infrastructure | Infrastructure state (emulator, headscale, daemon status) |
| summary | Aggregate counts: passed, failed, skipped, total |

### BugReport

Generated by explore agents when actual behavior diverges from spec. Used by the explore-fix-verify loop (local mode only).

| Field | Description |
|-------|-------------|
| id | Unique bug identifier |
| scenario_id | TestScenario that surfaced the bug |
| description | What went wrong (expected vs actual) |
| screenshot | Screenshot at point of failure |
| ui_hierarchy | DumpHierarchy XML at point of failure |
| severity | critical, major, minor |
| status | found, fixing, fixed, verified, regression |
| fix_files | List of source files modified to fix |
| fix_iteration | Which explore-fix-verify cycle fixed it |

### InfrastructureState

Tracks the state of test infrastructure components.

| Field | Description |
|-------|-------------|
| emulator_status | booting, ready, unresponsive, stopped |
| emulator_pid | Process ID |
| headscale_status | starting, ready, failed |
| headscale_url | Localhost URL (e.g., `http://127.0.0.1:18080`) |
| host_tailscale_ip | Tailscale IP assigned to host (100.x.x.x) |
| phone_tailscale_ip | Tailscale IP assigned to emulator |
| daemon_status | starting, listening, paired, stopped |
| daemon_socket | SSH_AUTH_SOCK path |
| mcp_server_status | starting, connected, disconnected |
| apk_installed | Boolean — whether debug APK is on emulator |

## Relationships

```
TestRun 1──* TestScenario
TestScenario 1──* BugReport (local mode only)
TestRun 1──1 InfrastructureState
```

## State Transitions

### TestScenario Lifecycle

```
pending → running → passed
pending → running → failed
pending → skipped (precondition not met)
```

### BugReport Lifecycle (local mode)

```
found → fixing → fixed → verified
found → fixing → fixed → regression → fixing (loop)
```

### InfrastructureState Setup Sequence

```
1. emulator: booting → ready
2. headscale: starting → ready
3. host tailscale: join → connected (IP assigned)
4. phone tailscale: join → connected (IP assigned)
5. daemon: starting → listening → paired (after pairing flow)
6. mcp_server: starting → connected
7. apk: not_installed → installed
```

Teardown is reverse order. All processes are killed on exit (trap handler).
