# Learnings

Discoveries, gotchas, and decisions recorded by the implementation agent across runs.

---

## T075 — CI pipeline

- `lewagon/wait-on-check-action@v1.3.4` does NOT poll for a check that hasn't been created yet — it fails immediately if the check name is not found. This causes a race condition when both CI and E2E workflows trigger on the same push event: the "CI Summary" check (which `needs:` four other jobs) doesn't exist in the GitHub Checks API until all its prerequisites complete.
- Fix: use `workflow_run` trigger (`types: [completed]`) instead of polling. This guarantees E2E only starts after the CI workflow finishes. Must use `github.event.workflow_run.head_sha` for checkout since `workflow_run` events default to the repo's default branch HEAD.
- `workflow_run` branch filter matches the branch that triggered the original workflow. For PRs, this is the head branch (not the base), so `branches: [develop, main]` won't match PR head branches from feature branches. This is acceptable if E2E on PRs is not required, or a hybrid approach is needed.
- The nixpkgs headscale module now enforces an assertion: `dns.nameservers.global must be set when overriding local DNS`. Any NixOS test using `services.headscale.settings.dns.base_domain` must also set `dns.nameservers.global` (e.g., `[ "127.0.0.1" ]`). Setting `magic_dns = false` avoids DNS complexity in tests.
- golangci-lint v2 with `errcheck` enabled catches all unchecked error returns including `defer x.Close()`, `go srv.Serve(lis)`, `tp.Shutdown(ctx)`, `fmt.Sscanf(...)`, etc. The diagnosis may undercount issues vs what the linter actually finds — always run the linter locally and iterate until 0 issues.
- The nixpkgs headscale module changed `tls_cert_path` and `tls_key_path` option types from `str` to `null or absolute path`. Empty strings `""` are no longer accepted — use `null` to disable TLS explicitly.
- CI failures can be "masked" by earlier step failures in the same job. When a preceding step (e.g., golangci-lint) is fixed, later steps (e.g., ktlint) may fail for the first time, revealing pre-existing issues like missing `android/gradlew`.
- In NixOS VM tests, node functions (`{ config, lib, pkgs, ... }:`) receive `pkgs` from the NixOS module system (base nixpkgs without overlays). If the test file receives an overlay-augmented `pkgs` from its outer scope (e.g., via `callTest`), destructuring `pkgs` in the node function shadows it. Use `{ config, lib, ... }:` to let the node body access the outer overlay-augmented `pkgs` for custom packages like `pkgs.nix-key`. Standard packages (`pkgs.age`, `pkgs.jq`) exist in both, so they work either way.
- When adding new NixOS VM test files that copy headscale boilerplate from existing tests, ALL three fixes must be applied: (1) nested `dns` attrset with `nameservers.global` and `magic_dns`, (2) `tls_cert_path = null` / `tls_key_path = null` instead of `""`, (3) no `pkgs` in node function args. The `signing-test.nix` file was a copy of `pairing-test.nix` infrastructure and needed the same three fixes applied across attempts 3-6.
- The `jaeger-all-in-one` package was completely removed from nixpkgs (not just renamed). Neither `pkgs.jaeger-all-in-one` nor `pkgs.jaeger` exist in nixpkgs-unstable as of rev `46db2e09e1d3f113a13c0d7b81e2f221c63b8ce9` (2026-03-22). All jaeger-related attributes are gone. Fix: add a custom `nix/jaeger.nix` package that fetches the pre-built Jaeger v2 binary from GitHub releases, and add it to the flake overlay. Jaeger v2 uses a single `jaeger` binary that defaults to all-in-one mode.
- NixOS module option defaults must not reference custom overlay packages (e.g., `default = pkgs.jaeger;`). The module's `pkgs` is base nixpkgs without overlays, so overlay-only packages will cause `attribute missing` errors. Instead, omit the default and require callers to set the package explicitly. Test files that receive overlay-augmented `pkgs` from `callTest` can set it there. This is the same class of issue as the pkgs-shadowing bug in test node functions (attempts 5-6), but applied to module option defaults (attempt 8).
- `tracing-e2e-test.nix` was yet another copy of the headscale boilerplate that needed the same triple-fix (DNS, TLS, pkgs-shadowing) — this is the third test file requiring it (after `pairing-test.nix` and `signing-test.nix`). When creating ANY new NixOS VM test with headscale, always apply all three fixes from the start. Consider extracting shared headscale infra into a common module to prevent this recurring pattern.
- Any change to `go.mod` or `go.sum` requires updating `vendorHash` in BOTH `nix/package.nix` and `nix/phonesim.nix`. The Nix build uses a fixed-output derivation for Go module vendoring, so a stale hash causes all dependent derivations (including all VM tests) to fail. The correct hash is shown in the Nix error output ("got: sha256-..."). **Important**: the two hashes are NOT guaranteed to be the same — `phonesim.nix` includes `test/phonesim` in its fileset which pulls in additional Go imports, producing a different vendor hash. Always update each hash independently using the value from its own build error.
- `systemctl --user -M <user>@ cat <unit>` does NOT work — systemd returns "Cannot remotely cat units." The `cat` verb doesn't support the `-M` flag with `--user`. Use `cat /etc/systemd/user/<unit>` instead for NixOS-installed user units (placed there by `systemd.user.services`). Other verbs like `start`, `stop`, `show`, and `is-system-running` work fine with `-M --user`.
- Jaeger v2 (v2.16.0) started without `--config` uses a default exporter pipeline that fails with "traces export: context deadline exceeded: rpc error: code = DeadlineExceeded". The default pipeline tries to forward traces via gRPC to a non-existent endpoint rather than storing in-memory. Fix: provide an explicit YAML config file via `--config` that defines the full pipeline: `jaeger_storage` extension with `memstore` backend, `jaeger_storage_exporter`, `jaeger_query` extension, `otlp` receiver. Do NOT include a `batch` processor — even with a correct `--config`, `batch:` (which YAML parses as `batch: null`) causes `jaeger_storage_exporter` to fail with the same "context deadline exceeded" gRPC error. For low-volume dev/test scenarios, traces should flow directly from receiver to exporter without batching.
- The `jaeger_storage_exporter` is a custom Jaeger v2 "Development component" that does NOT inherit the standard OTel Collector `exporterhelper` config fields. Setting `timeout`, `sending_queue`, or `retry_on_failure` on it causes a startup crash: `'storageexporter.Config' has invalid keys: sending_queue, timeout`. The only accepted config key is `trace_storage`. On resource-constrained VMs where the exporter hits "context deadline exceeded", the workaround is to increase the test polling timeout (e.g., `wait_until_succeeds` timeout=120) rather than tuning the exporter config.
- Headscale v0.28.0 requires a non-empty DERP map at startup. Setting `derp.urls = []; derp.paths = [];` prevents internet fetches but causes `"initial DERPMap is empty"` crash. Fix: provide a minimal static DERP map JSON file via `environment.etc."headscale/derp.json"` containing one region (RegionID 900, STUNPort -1, DERPPort 0) and set `derp.paths = [ "/etc/headscale/derp.json" ];`. The DERP server doesn't need to be functional — headscale just needs a non-empty map to pass startup validation. Tailscale clients communicate over the local VLAN without DERP relay. This is yet another headscale module gotcha alongside DNS nameservers, TLS path types, and pkgs-shadowing.
- The `jaeger_storage_exporter` "Development component" is fundamentally unreliable on 1-CPU CI VMs — it hits an unconfigurable 5s context deadline that causes "traces export: context deadline exceeded" consistently. After 8+ failed attempts tuning config, the workaround is to add a `debug` exporter (verbosity: detailed) to the pipeline and verify traces via `journalctl` instead of the Jaeger query API.
- The CI workflow's `cancel-in-progress: true` concurrency setting will kill long-running NixOS VM test jobs (~15-20 min) if any new commit is pushed to the same branch during execution. This caused attempts 23 and 26 to be cancelled without producing any test results. Fix: set `cancel-in-progress: false` so runs complete even when newer commits are pushed. Trade-off: uses more CI minutes but prevents the repeated cancellation loop.
- NixOS VM test scripts are linted by pyflakes before execution. Any unused Python imports (`import time`, `import os`, etc.) cause a build failure for the test driver derivation. When adding `import time` (or similar) to one test file, verify it's actually used. When fixing an unused import in one test file, audit ALL test files (`nix/tests/*-test.nix`) for the same issue — the same import was often copy-pasted to multiple files (e.g., attempts 19-20 fixed `import time` in pairing-test.nix then signing-test.nix).
- Headscale v0.28.0 changed the `--user` flag on `preauthkeys create` from accepting a username string to requiring a numeric user ID (uint). `headscale preauthkeys create --user nixkey-test` silently fails with exit code 1. Fix: after `headscale users create <name>`, retrieve the numeric ID via `headscale users list -o json | jq -r '.[0].id'` and pass it as `--user <id>`. Also avoid `2>/dev/null` on preauthkeys commands so errors are visible.
- A static DERP map with `hostname: 127.0.0.1` and `derpport: 0` (defaults to 443) passes headscale startup validation but is non-functional — nothing listens on 127.0.0.1:443 in VM tests. Without a working DERP relay, tailscale nodes can't exchange disco messages for WireGuard tunnel setup, even on the same VLAN. Fix: enable headscale's embedded DERP server (`derp.server.enabled = true`, `derp.server.ipv4 = "192.168.1.1"`, `derp.server.automatically_add_embedded_derp_region = true`) and update the static DERP map's hostname/ipv4 to `192.168.1.1` with `derpport: 8080` (matching headscale's listen port). Also increase ping timeouts to 120s since WireGuard tunnel setup via DERP relay can take time on slow CI VMs.
- When a NixOS VM test imports `nixKeyModule` with `services.nix-key.enable = true` and enables user lingering, the `nix-key-agent` systemd user service auto-starts on boot **before** the test writes `devices.json`. Manual `su - testuser -c 'nix-key daemon ...'` starts fail silently because the config.json generated by the module omits `controlSocketPath` (it defaults to `""` and uses `lib.optionalAttrs`), while the systemd service gets it via `EnvironmentFile`. Result: `ssh-add -L` connects to the auto-started empty-registry daemon and returns "no identities". Fix: (1) stop the systemd service before manual daemon starts with `systemctl --user -M testuser@ stop nix-key-agent.service`, and (2) set `NIXKEY_CONTROL_SOCKET_PATH` env var in all manual daemon start commands.
- In NixOS VM tests, `pkill -f phonesim` inside `phone.succeed("pkill -f phonesim || true")` self-matches: `pkill` sends SIGTERM to the shell process running the command itself, resulting in exit code 143 (128+15) before `|| true` can execute. Fix: use `phone.execute("pkill phonesim")` — `execute()` ignores exit codes, and dropping `-f` avoids matching the full command line (which includes the pkill invocation itself).
- The `nix-key daemon` command was a stub that printed "daemon: not yet implemented" and exited immediately, never creating the SSH agent socket. This caused all VM tests that start the daemon (signing-test, tracing-e2e-test) to fail with a 30-second timeout waiting for the agent socket. The stub was left over from T002 (skeleton), and the daemon should have been wired up when the agent/backend/control components were implemented in phases 5-6. The fix: wire the daemon command to load config, create device registry, start GRPCBackend with a production dialer (plain gRPC when no certs, mTLS when certs present), start agent server, and start control server with graceful shutdown.
- In tracing-e2e tests, `wait_until_succeeds("curl ... | jq '.data | length'")` succeeds immediately with "0" when Jaeger returns an empty trace list — the command exits 0 even though no traces exist. Fix: use `jq -e '.data | length > 0'` so `jq` exits non-zero when the expression is false, causing `wait_until_succeeds` to retry until traces actually appear.
- The Jaeger OTLP receiver in module.nix bound to `127.0.0.1:4317` by default, which is unreachable from the phone VM via Tailscale IP. In the tracing-e2e test, phonesim sends traces to `host_ts_ip:4317` (e.g., `100.64.0.2:4317`), which gets RST because Jaeger isn't listening on that interface. Fix: bind OTLP endpoints to `0.0.0.0` so they're accessible from all interfaces including Tailscale.
- When a NixOS VM test derivation is cached as failed, subsequent runs with the same derivation hash will use the cached failure — even if the underlying issue was fixed in the source tree (e.g., the fix didn't change the test's Nix inputs). Any change to a transitive input (e.g., module.nix, which signing-test.nix imports) will invalidate the cache.
- When killing a phonesim process that was started with `-sign-delay` (e.g., 60s), `pkill phonesim` (SIGTERM) may fail silently because the process is blocked mid-request in the delayed sign handler. Use `pkill -9 phonesim` (SIGKILL) to reliably kill it. This applies specifically to Phase 9 of signing-test.nix where the sign-delay phonesim from Phase 8 needs to be stopped.
- The `tracing.Init()` function must call `otel.SetTextMapPropagator(propagation.TraceContext{})` when initializing a real OTLP exporter. Without this, the global propagator is a no-op, and `otelgrpc.NewClientHandler()` cannot inject W3C `traceparent` headers into gRPC metadata. This breaks distributed tracing (host→phone span linking) even though local spans are created and exported. The phonesim's tracing init (`test/phonesim/main.go`) already sets this propagator correctly; the daemon's `internal/tracing/tracing.go` was missing it.
- The default `sdktrace.WithBatcher(exp)` has a 5-second batch timeout. In VM tests where the sign operation completes quickly and traces are queried shortly after, reducing the batch timeout (e.g., `WithBatchTimeout(2*time.Second)`) improves reliability of trace appearance in Jaeger.
- In NixOS VMs, `localhost` resolves to both `::1` (IPv6) and `127.0.0.1` (IPv4) via `/etc/hosts`. When the Go gRPC OTLP exporter connects to `localhost:4317`, gRPC's DNS resolver (default scheme in `grpc.NewClient` since gRPC-go v1.67) may try IPv6 first. If Jaeger only listens on `0.0.0.0:4317` (IPv4), the IPv6 connection fails and the batch exporter may drop spans during the resolver fallback window. Fix: use `127.0.0.1:4317` instead of `localhost:4317` for OTLP endpoints in NixOS module config to avoid IPv6 resolution entirely.
