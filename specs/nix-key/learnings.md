## T075 â€” CI pipeline

- `lewagon/wait-on-check-action@v1.3.4` does NOT poll for a check that hasn't been created yet â€” it fails immediately if the check name is not found. This causes a race condition when both CI and E2E workflows trigger on the same push event: the "CI Summary" check (which `needs:` four other jobs) doesn't exist in the GitHub Checks API until all its prerequisites complete.
- Fix: use `workflow_run` trigger (`types: [completed]`) instead of polling. This guarantees E2E only starts after the CI workflow finishes. Must use `github.event.workflow_run.head_sha` for checkout since `workflow_run` events default to the repo's default branch HEAD.
- `workflow_run` branch filter matches the branch that triggered the original workflow. For PRs, this is the head branch (not the base), so `branches: [develop, main]` won't match PR head branches from feature branches. This is acceptable if E2E on PRs is not required, or a hybrid approach is needed.
- The nixpkgs headscale module now enforces an assertion: `dns.nameservers.global must be set when overriding local DNS`. Any NixOS test using `services.headscale.settings.dns.base_domain` must also set `dns.nameservers.global` (e.g., `[ "127.0.0.1" ]`). Setting `magic_dns = false` avoids DNS complexity in tests.
- golangci-lint v2 with `errcheck` enabled catches all unchecked error returns including `defer x.Close()`, `go srv.Serve(lis)`, `tp.Shutdown(ctx)`, `fmt.Sscanf(...)`, etc. The diagnosis may undercount issues vs what the linter actually finds â€” always run the linter locally and iterate until 0 issues.
- The nixpkgs headscale module changed `tls_cert_path` and `tls_key_path` option types from `str` to `null or absolute path`. Empty strings `""` are no longer accepted â€” use `null` to disable TLS explicitly.
- CI failures can be "masked" by earlier step failures in the same job. When a preceding step (e.g., golangci-lint) is fixed, later steps (e.g., ktlint) may fail for the first time, revealing pre-existing issues like missing `android/gradlew`.
- In NixOS VM tests, node functions (`{ config, lib, pkgs, ... }:`) receive `pkgs` from the NixOS module system (base nixpkgs without overlays). If the test file receives an overlay-augmented `pkgs` from its outer scope (e.g., via `callTest`), destructuring `pkgs` in the node function shadows it. Use `{ config, lib, ... }:` to let the node body access the outer overlay-augmented `pkgs` for custom packages like `pkgs.nix-key`. Standard packages (`pkgs.age`, `pkgs.jq`) exist in both, so they work either way.
- When adding new NixOS VM test files that copy headscale boilerplate from existing tests, ALL three fixes must be applied: (1) nested `dns` attrset with `nameservers.global` and `magic_dns`, (2) `tls_cert_path = null` / `tls_key_path = null` instead of `""`, (3) no `pkgs` in node function args. The `signing-test.nix` file was a copy of `pairing-test.nix` infrastructure and needed the same three fixes applied across attempts 3-6.
- The `jaeger-all-in-one` package was completely removed from nixpkgs (not just renamed). Neither `pkgs.jaeger-all-in-one` nor `pkgs.jaeger` exist in nixpkgs-unstable as of rev `46db2e09e1d3f113a13c0d7b81e2f221c63b8ce9` (2026-03-22). All jaeger-related attributes are gone. Fix: add a custom `nix/jaeger.nix` package that fetches the pre-built Jaeger v2 binary from GitHub releases, and add it to the flake overlay. Jaeger v2 uses a single `jaeger` binary that defaults to all-in-one mode.
- NixOS module option defaults must not reference custom overlay packages (e.g., `default = pkgs.jaeger;`). The module's `pkgs` is base nixpkgs without overlays, so overlay-only packages will cause `attribute missing` errors. Instead, omit the default and require callers to set the package explicitly. Test files that receive overlay-augmented `pkgs` from `callTest` can set it there. This is the same class of issue as the pkgs-shadowing bug in test node functions (attempts 5-6), but applied to module option defaults (attempt 8).
- `tracing-e2e-test.nix` was yet another copy of the headscale boilerplate that needed the same triple-fix (DNS, TLS, pkgs-shadowing) â€” this is the third test file requiring it (after `pairing-test.nix` and `signing-test.nix`). When creating ANY new NixOS VM test with headscale, always apply all three fixes from the start. Consider extracting shared headscale infra into a common module to prevent this recurring pattern.
- Any change to `go.mod` or `go.sum` requires updating `vendorHash` in BOTH `nix/package.nix` and `nix/phonesim.nix`. The Nix build uses a fixed-output derivation for Go module vendoring, so a stale hash causes all dependent derivations (including all VM tests) to fail. The correct hash is shown in the Nix error output ("got: sha256-..."). **Important**: the two hashes are NOT guaranteed to be the same â€” `phonesim.nix` includes `test/phonesim` in its fileset which pulls in additional Go imports, producing a different vendor hash. Always update each hash independently using the value from its own build error.
- `systemctl --user -M <user>@ cat <unit>` does NOT work â€” systemd returns "Cannot remotely cat units." The `cat` verb doesn't support the `-M` flag with `--user`. Use `cat /etc/systemd/user/<unit>` instead for NixOS-installed user units (placed there by `systemd.user.services`). Other verbs like `start`, `stop`, `show`, and `is-system-running` work fine with `-M --user`.
- Jaeger v2 (v2.16.0) started without `--config` uses a default exporter pipeline that fails with "traces export: context deadline exceeded: rpc error: code = DeadlineExceeded". The default pipeline tries to forward traces via gRPC to a non-existent endpoint rather than storing in-memory. Fix: provide an explicit YAML config file via `--config` that defines the full pipeline: `jaeger_storage` extension with `memstore` backend, `jaeger_storage_exporter`, `jaeger_query` extension, `otlp` receiver. Do NOT include a `batch` processor â€” even with a correct `--config`, `batch:` (which YAML parses as `batch: null`) causes `jaeger_storage_exporter` to fail with the same "context deadline exceeded" gRPC error. For low-volume dev/test scenarios, traces should flow directly from receiver to exporter without batching.
